Cursorの未修正脆弱性、リポジトリを開くだけでRCE可能　AIコーディング環境の安全性に再び懸念

  • Cursorの脆弱性が7か月未修正で残る
  • リポジトリを開くだけでRCEに至りうる
  • AI開発ツール全般の安全性が問われる

AIコーディング環境のCursorで、リポジトリを開いただけで任意コード実行につながる脆弱性の詳細が公開された。研究者が12月に報告してから7か月たっても修正されておらず、poisoned repository攻撃で悪用できるとされる。[1][2]

A vulnerability in Cursor can lead to arbitrary code execution simply by opening a repository. Researchers say it was reported months ago but remains unfixed, renewing concerns about the security of AI coding tools.[1][2]

AIコーディング支援ツールのCursorで、Windows版に関わる深刻な脆弱性の詳細が公開された。ポイントは、利用者が「リポジトリを開く」だけで任意コード実行につながりうることだ。研究者側は12月に報告していたが、7か月後の現在も修正されていないという。[1][2]

攻撃の筋書きは比較的シンプルだ。悪意ある攻撃者がリポジトリ内に偽の「git.exe」を仕込んでおき、開発者がそのリポジトリをCursorで開くと、意図しないコードが実行される可能性がある。つまり、明示的にプログラムを起動しなくても、開発作業の入口そのものが攻撃面になってしまう。[2]

この種の問題が厄介なのは、AI支援IDEやエージェント型開発環境が、従来のエディタよりも広い権限や自動化を前提にしている点にある。開発者の作業効率を上げる仕組みほど、ひとたび信頼境界を越えると被害が大きくなりやすい。Cursorの件は、便利さと安全性が常にトレードオフではないにせよ、設計段階での防御が不十分だと深刻なRCEにつながることを改めて示した。[1][2]

同じくAI開発ツールの周辺では、Claude Code、GitHub Copilot、LangChainといった製品・フレームワークの利用も広がっている。今回のような事案は、単一製品の不具合にとどまらず、開発者ツール全体で「ローカルファイル」「外部リポジトリ」「自動実行」の組み合わせをどう制御するかという共通課題を突きつける。AIがコードを書く時代ほど、開く前の検査と権限分離の徹底が重要になる。[1][2]
