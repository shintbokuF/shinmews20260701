Cursor Windows版に未修正の任意コード実行脆弱性、開くだけで危険

  • Cursor Windows版に未修正のRCE穴が残る
  • リポジトリを開くだけで起動時実行が起きうる
  • AI開発ツール全体の信頼設計が揺らぐ

AIコードエディタCursorのWindows版で、リポジトリを「開くだけ」で任意コード実行につながる脆弱性が、報告から7か月以上たっても未修正のままだと報じられた。研究者は詳細を公開し、悪性のgit.exeを仕込んだ“poisoned repository”で起動時にコードを走らせられる可能性を示している。AI開発ツールでは、承認プロンプトや人間の確認を前提にした設計が突破される問題も相次いでおり、単独の製品不具合ではなく業界全体の課題として浮かび上がっている。[1][3][5][6]

Cursor’s Windows version reportedly still contains an unfixed vulnerability that can lead to arbitrary code execution simply by opening a repository. The case highlights broader security weaknesses in AI coding tools and agent workflows, where human approval and trust boundaries can be manipulated.[1][3][5][6]

AIコードエディタCursorのWindows版をめぐり、「リポジトリを開くだけ」で任意コード実行につながる脆弱性が、報告から7か月以上たっても未修正だと伝えられた。研究者は今回、その詳細を公開し、悪意あるリポジトリを読み込ませるだけで危険なコードが走る可能性を改めて示している[1][6]。

ポイントは、従来の“実行”操作ではなく、開発者が日常的に行う“open repository”の段階に攻撃面があることだ。Dark Readingは、研究者が12月にCursorへ報告したものの、いまも脆弱性が残るとし、リポジトリ内に悪性の「git.exe」を置くことで、プロジェクトを開いた開発者の環境で実行を狙えると説明している[6]。

この種の問題はCursor単体に限られない。Wizの研究として報じられた「GhostApproval」では、AI開発ツールの承認プロセスそのものが人間をだます形で悪用され、サンドボックスからの脱出を許す設計上の欠陥が6つの主要なAIコーディングアシスタントにまたがっていたという[3][5]。つまり、AI支援の開発環境では、モデルや自動化の賢さよりも、最終確認を担う人間の判断や承認導線が攻撃対象になりやすい[3][5]。

Cursorの件が示すのは、生成AI時代の開発ツールでは「便利さ」と「安全な初期挙動」の両立が難しいという現実だ。GitHub CopilotやClaude Code、LangChain系のエージェントフレームワークも含め、コード生成だけでなく、リポジトリ読み込み、ツール実行、権限承認の一連の流れをどこまで厳格に制御できるかが、今後の競争力と信頼性を左右する。今回の未修正問題は、AI開発ツールのセキュリティが“後付け”では済まされない段階に来ていることを、あらためて突きつけた[1][3][5][6]。
